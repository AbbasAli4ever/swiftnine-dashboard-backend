# Task Description & Attachments — Frontend Integration Guide

> **Audience:** Frontend engineers integrating rich task descriptions and file attachments.
> **Last updated:** 2026-04-30
> **Backend author:** Platform team

---

## Feature Readiness

| Capability | Status | Notes |
|---|---|---|
| Rich text description (Tiptap JSON) | ✅ Ready | `descriptionJson` field on task |
| Plain text description (legacy) | ✅ Ready | `description` field (simple string) |
| Full-text search on description | ✅ Ready | Auto-extracted into `descriptionPlaintext` |
| File attachment upload (S3) | ✅ Ready | Presigned PUT flow |
| List task attachments with view URLs | ✅ Ready | Presigned GET URLs, 15-min expiry |

## Authentication

All endpoints require a Bearer JWT in the `Authorization` header and the workspace ID in the `x-workspace-id` header.

```
Authorization: Bearer <accessToken>
x-workspace-id: <workspaceId>
```

## Part 1 — Rich Text Description

### Overview

The task description is stored as a Tiptap/ProseMirror JSON document in the `descriptionJson` field. This mirrors the exact format used by doc content (`contentJson`). Use the same Tiptap editor instance you already use for docs.

The backend automatically extracts `descriptionPlaintext` from the JSON for search — you never send or receive this field directly.

### Creating a task with a rich description

```
POST /api/v1/projects/:projectId/lists/:listId/tasks
x-workspace-id: <workspaceId>
Authorization: Bearer <token>
Content-Type: application/json
```

```json
{
  "title": "Implement login page",
  "statusId": "uuid-of-status",
  "priority": "HIGH",
  "descriptionJson": {
    "type": "doc",
    "content": [
      {
        "type": "paragraph",
        "content": [{ "type": "text", "text": "Build the login page using NextAuth." }]
      }
    ]
  }
}
```

### Updating a task description

Send only the fields you want to change. All fields are optional.

```
PATCH /api/v1/tasks/:taskId
x-workspace-id: <workspaceId>
Authorization: Bearer <token>
Content-Type: application/json
```

```json
{
  "descriptionJson": {
    "type": "doc",
    "content": [
      {
        "type": "paragraph",
        "content": [{ "type": "text", "text": "Updated description." }]
      },
      {
        "type": "bulletList",
        "content": [
          { "type": "listItem", "content": [{ "type": "paragraph", "content": [{ "type": "text", "text": "Step 1" }] }] },
          { "type": "listItem", "content": [{ "type": "paragraph", "content": [{ "type": "text", "text": "Step 2" }] }] }
        ]
      }
    ]
  }
}
```

To **clear** the description, send `null`:

```json
{ "descriptionJson": null }
```

### Reading the description

`GET /api/v1/tasks/:taskId` returns both fields:

```json
{
  "data": {
    "id": "...",
    "title": "Implement login page",
    "description": null,
    "descriptionJson": {
      "type": "doc",
      "content": [...]
    },
    "descriptionPlaintext": "Build the login page using NextAuth.",
    ...
  }
}
```

- Render `descriptionJson` in your Tiptap editor.
- `description` is the legacy plain-text field — ignore it unless you explicitly need plain text.
- Never send `descriptionPlaintext` — it is read-only, auto-generated by the backend.

### Size limit

`descriptionJson` is capped at **1 MB** (same as doc content). The server returns `413 Payload Too Large` if exceeded.

## Part 2 — File Attachments

File uploads use a **two-step presigned S3 flow** — the same pattern used for doc attachments.

```
Step 1: GET presigned upload URL  →  POST /api/v1/attachments/presign
Step 2: Upload file directly to S3 using the presigned URL (PUT, no auth header)
Step 3: Register the attachment record  →  POST /api/v1/attachments
```

### Step 1 — Get a presigned upload URL

```
POST /api/v1/attachments/presign
Authorization: Bearer <token>
Content-Type: application/json
```

```json
{
  "taskId": "uuid-of-task",
  "fileName": "screenshot.png",
  "mimeType": "image/png"
}
```

**Response:**

```json
{
  "data": {
    "uploadUrl": "https://s3.amazonaws.com/bucket/swiftnine/docs/app/attachments/task-<taskId>/uuid-screenshot.png?X-Amz-...",
    "s3Key": "swiftnine/docs/app/attachments/task-<taskId>/uuid-screenshot.png",
    "expiresAt": "2026-04-30T14:15:00.000Z"
  }
}
```

> The presigned URL expires in **15 minutes**. Start the upload immediately.

### Step 2 — Upload directly to S3

```
PUT <uploadUrl>
Content-Type: image/png   ← must match the mimeType you sent in Step 1

<file binary>
```

Important:
- Do **not** include `Authorization` — the URL is already signed.
- Set `Content-Type` to the same MIME type you sent in Step 1.
- No multipart encoding — send the raw file body.

### Step 3 — Register the attachment

After S3 confirms the upload (HTTP 200), call:

```
POST /api/v1/attachments
Authorization: Bearer <token>
Content-Type: application/json
```

```json
{
  "taskId": "uuid-of-task",
  "memberId": "uuid-of-current-user",
  "s3Key": "swiftnine/docs/app/attachments/task-<taskId>/uuid-screenshot.png",
  "fileName": "screenshot.png",
  "mimeType": "image/png",
  "fileSize": 204800
}
```

> `memberId` is the current user's UUID (or workspace member UUID — both are accepted).
> `fileName`, `mimeType`, `fileSize` are optional if you trust the backend to resolve them from S3 metadata, but providing them avoids an extra S3 `HeadObject` call.

**Response:**

```json
{
  "data": {
    "id": "attachment-uuid",
    "s3Key": "swiftnine/docs/app/attachments/task-.../uuid-screenshot.png",
    "fileName": "screenshot.png",
    "mimeType": "image/png",
    "fileSize": 204800,
    "createdAt": "2026-04-30T14:00:00.000Z"
  }
}
```

### Listing attachments (with view URLs)

```
POST /api/v1/attachments/view
Authorization: Bearer <token>
Content-Type: application/json
```

```json
{
  "taskId": "uuid-of-task",
  "memberId": "uuid-of-current-user"
}
```

**Response:**

```json
{
  "data": [
    {
      "id": "attachment-uuid",
      "fileName": "screenshot.png",
      "mimeType": "image/png",
      "s3Key": "...",
      "fileSize": 204800,
      "url": "https://s3.amazonaws.com/...?X-Amz-...",
      "expiresAt": "2026-04-30T14:15:00.000Z"
    }
  ]
}
```

- `url` is a presigned GET URL valid for **15 minutes**.
- Re-call this endpoint when URLs expire — do not cache them longer than 14 minutes.
- Use `url` directly in `<img src>`, `<a href>`, or as a download link.

### Deleting an attachment

```
DELETE /api/v1/attachments
Authorization: Bearer <token>
Content-Type: application/json
```

```json
{
  "taskId": "uuid-of-task",
  "memberId": "uuid-of-current-user",
  "s3Key": "swiftnine/docs/app/attachments/task-.../uuid-screenshot.png"
}
```

**Response:**

```json
{
  "data": {
    "id": "attachment-uuid",
    "s3Key": "..."
  }
}
```

> This is a soft delete — the file remains in S3 but the record is hidden and view URLs will no longer be issued for it.

## Part 3 — Recommended UI Flow

```
User opens task detail
  ├─ Load task  →  GET /tasks/:taskId
  │    ├─ Hydrate Tiptap editor with descriptionJson (or empty doc if null)
  │    └─ Load attachments  →  POST /attachments/view
  │
  ├─ User edits description
  │    └─ Autosave (debounce ~1s)  →  PATCH /tasks/:taskId { descriptionJson }
  │
  └─ User uploads a file
       ├─ 1. POST /attachments/presign  →  get uploadUrl + s3Key
       ├─ 2. PUT <uploadUrl>            →  upload binary to S3
       ├─ 3. POST /attachments          →  register record
       └─ 4. Append to local attachment list (no need to re-fetch)
```

### Autosave recommendation

- Debounce description saves by ~1 second after the user stops typing.
- Show a subtle "Saving..." / "Saved" indicator.
- On network failure, queue the save and retry — do not discard the editor state.

### File upload UX

- Show a progress indicator during the S3 PUT (Step 2) — that is where the actual upload time is.
- Steps 1 and 3 are fast API calls; only Step 2 is slow (file transfer).
- After Step 3 succeeds, append the returned attachment to your local list immediately — no re-fetch needed.

## Error Reference

| Status | Meaning |
|--------|---------|
| 400 | Bad request — invalid JSON shape, only one upload scope allowed |
| 401 | Missing or invalid access token |
| 403 | Actor is not the workspace member, or lacks permission |
| 404 | Task, member, or attachment not found |
| 413 | `descriptionJson` exceeds 1 MB limit |
| 500 | S3 bucket not configured, or failed to read S3 metadata |
